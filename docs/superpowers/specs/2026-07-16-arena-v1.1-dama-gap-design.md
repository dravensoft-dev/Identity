# Arena v1.1 — closing the DAMA component gap

**Date:** 2026-07-16 · **Status:** approved, ready for planning · **Supersedes:** the proposals in `components-catalog.md`

`components-catalog.md` is an audit: it lists what Arena lacks against DAMA's needs and explicitly marks every item "proposed, not approved" with four open design decisions. This spec resolves those decisions and specifies what gets built. Where it contradicts the catalog, this spec wins; each divergence is recorded with its reason.

## Scope

**In (this minor, v1.1.0):** categorical palette tokens, breakpoint tokens, the palette-swap restructure, chart family, `StatCard`, `PageHead`, `Spinner`, `ThemeToggle`, `Table` responsive mode, `Input` date/time support.

**Out (next minor):** `Calendar` / `Scheduler` (item 3). Large, and it slips a release on its own.

**Out (permanently):** `List` (item 10) — both DAMA usages are better served by `Shell`'s nav and plain markup. `SegmentedControl` (item 9) — only if the release has room; not planned. `DatePicker` / `TimePicker` — the migration needs native date inputs, not a custom picker.

## Decisions

Seven decisions were settled during design. The four the catalog flagged, plus three the work surfaced.

| # | Decision | Rationale |
|---|---|---|
| 1 | **Charts are dependency-free SVG**, not Chart.js | A `<canvas>` cannot inherit CSS, which is the *only* reason the catalog's `ChartPalette` contract exists. SVG reads `var(--color-cat-N)` directly and is theme-reactive with no code. Also fits the copy-in kit, which has no `package.json` to add a peer to. |
| 2 | **`StatCard` delta separates direction from sentiment** | The catalog's `sign: 'up' \| 'down'` conflates them. Revenue down is bad; latency down is good. A component cannot know, so the product declares it. |
| 3 | **Delta pills are outline, both signs** | Keeps filled red exclusive to `ConfirmDialog`'s final confirmation, which is what gives it weight. Destructive affordances are rare and should not compete for attention; a data pill has no business spending that signal. Applies to `positive` too, so the pair reads as a family. |
| 4 | **The palette is a documented default, not a mandate** | Arena's identity lives in shape — danger-as-outline, no gradients, the surface scale, the hairline border, the warm shadow — all of which survive a palette swap. Crimson and gold are Dravensoft's skin. The architecture already supported this; it was never declared. |
| 5 | **Categorical ramp: 8 slots, cool arc, fixed order** | Derived by enumeration against the dataviz validator, not chosen by eye. See below. |
| 6 | **Responsive is measured per container, not per viewport** | Inline styles cannot hold media queries, so this is JS regardless. Given that, measuring the container is no harder than the viewport and is correct in more cases: a `Table` in a narrow card should go card-mode on a wide monitor. |
| 7 | **The validator ships in the repo** | Decision 4 promises the palette can be swapped. A promise with no way to re-check the gates is not a promise. |

## Token layer

### Palette extraction

- **New `tokens/palette.css`** — the Dravensoft skin only: the `--color-*` blocks for `:root` (dark) and `.arena-light`, plus the categorical ramp. Nothing else.
- **`tokens/colors.css` keeps the structure** — the compatibility layer (`--bg`, `--crimson`, `--danger`, `--mute`…) and the `color-mix` derivations. It reads tokens; it never defines skin values.
- **`styles.css`** imports `palette.css` before `colors.css`.

A consumer swaps one file. The public swap surface is the `--color-*` set plus `--color-cat-*`; everything else derives.

### Categorical ramp

Eight slots, authored per theme, **fixed order, never cycled**. A ninth series folds to "Other", small multiples, or direct labels — it is never a generated hue.

| Slot | Name | Hue | Dark | Light |
|---|---|---|---|---|
| 1 | forest | 136° | `#3c7b0a` | `#397804` |
| 2 | indigo | 264° | `#3b63be` | `#264ba4` |
| 3 | green | 152° | `#0a924b` | `#0a924b` |
| 4 | violet | 288° | `#6a59bc` | `#523e9f` |
| 5 | cyan | 216° | `#00a3c0` | `#008fa9` |
| 6 | purple | 312° | `#884da9` | `#6e328d` |
| 7 | teal | 184° | `#00a99a` | `#009487` |
| 8 | orchid | 328° | `#984697` | `#7c2b7b` |

**How it was derived.** Candidate hues were filtered to those that clear the chroma floor *and* 3:1 contrast against the real chart surface (`--color-base-200`, `#1d1715` dark / `#f7f4ef` light) in both themes; the whole crimson→gold warm arc was banned; the eight hues were then picked to maximize distance from the status colors, and the slot order derived by enumerating orderings against the validator. Chroma is capped at OKLCH C ≤ 0.15 so the ramp sits in Arena's own register (crimson 0.177, gold 0.100, success 0.128) rather than reading as neon.

**Measured (authoritative, both themes pass every hard gate):**

| Gate | Dark | Light | Bar |
|---|---|---|---|
| CVD separation (adjacent, OKLab ΔE×100) | 13.3 | 16.4 | target ≥ 8 |
| Normal-vision floor | 20.5 | 22.1 | hard floor ≥ 15 |
| Contrast vs surface | all 8 ≥ 3:1 | all 8 ≥ 3:1 | ≥ 3:1 |
| Lightness band | all inside | all inside | per-mode band |
| Chroma floor | all ≥ 0.1 | all ≥ 0.1 | ≥ 0.10 |

No relief rule is needed — unlike the dataviz skill's own reference palette, which has three light-mode slots below 3:1.

**Brand clearance (ΔE vs the ramp's closest slot):** crimson 17.0, gold 18.0, error 19.6, warning 26.3 — all above the 15 bar. This was the goal: the ramp cannot be mistaken for the brand or for an error.

**Accepted collision:** success 6.0, info 7.8. This is structural, not an oversight. Eight slots need ~126° of arc; banning the red family leaves green/cyan/blue/violet, which is where success (156°) and info (250°) live, and guarding them as hard as the brand leaves only ~76°. **A ramp can be clear of the brand or clear of status, not both.** Clear of the brand is correct: brand colors carry identity everywhere, while status colors always ship with an icon and a label (`Alert`, `Toast`, `Badge`) and never appear as a bare fill. This is the dataviz skill's own sanctioned mitigation, and its reference palette accepts tighter collisions (4.8) than ours.

### Breakpoints

`--bp-sm: 480px`, `--bp-md: 768px`, `--bp-lg: 1024px` in `tokens/spacing.css`. These are **shared values read by JS**, not media queries — inline styles cannot hold a media query. `768px` matches what DAMA hardcodes.

### Chart chrome

Charts consume existing tokens. No new chrome tokens.

> **Catalog correction.** The catalog says the grid "maps to the divider/border token; there is no `--chart-grid` in Arena today — add it or alias to `--line`." **`--line` does not exist** in `tokens/colors.css` (only `--line-strong` and `--border`). Grid uses `--border`; axis and baseline use `--line-strong`; labels use `--text-muted`.

## Hooks

**`useContainerWidth()`** — returns `[ref, width]` via `ResizeObserver`. Width is `null` until first measure; consumers render the wide layout on `null` so SSR and first paint are not the narrow branch. Used by `Table`, and by `Calendar` next minor.

Lives at **`use-container-width.js` in the repo root**, beside `theme.js` and `jsx-loader.js` — the established place for a shared runtime helper. No `hooks/` directory is created for one file. Because it is a new copy-in artifact, `README.md`'s copy list must gain it alongside `tokens/`, `assets/` and `styles.css`.

## Charts

Four components: `ChartCard`, `BarChart`, `LineChart`, `DoughnutChart`.

### What is dropped from the catalog

`ChartPalette`, `readChartPalette`, `seriesColor`, `withAlpha` and the hardcoded fallback palette **are all deleted from the design**. They exist only to work around canvas's inability to inherit CSS. With SVG, `fill="var(--color-cat-3)"` is theme-reactive for free, and the "one sanctioned place for literal colors" the catalog carves out is not needed.

`CHART_SERIES_ORDER` (`primary → success → warning → danger → textMuted`) **is deleted**. It uses status colors as series colors — a series painted `--danger` reads as an error. This is exactly the bug the catalog warns about in item 11, and it had leaked into item 1. The dataviz skill reserves status colors explicitly.

### Color contract

- **Identity** → the categorical ramp by slot index, assigned in order, never cycled.
- **Meaning** → `tone?: 'success' | 'warning' | 'danger' | 'info'`, opt-in per series, for when a series *is* a state (error rate, pass/fail).
- Never both in one chart.
- Text — values, labels, legends — wears text tokens, never the series color.

### API

```ts
export type CatSlot = 1|2|3|4|5|6|7|8;
export type SeriesTone = 'success'|'warning'|'danger'|'info';

export interface ChartCardProps {
  title?: string;               // uppercase muted label, NOT an h*
  actions?: React.ReactNode;
  children: React.ReactNode;
}
export interface BarChartProps {
  labels: string[];
  values: number[];
  seriesLabel?: string;
  slot?: CatSlot;               // one color for all bars, default 1
  slots?: CatSlot[];            // per-bar override, wins over slot
  tone?: SeriesTone;            // semantic override; mutually exclusive with slot/slots
  valueFormatter?: (value: number) => string;
}
export interface LineChartProps extends Omit<BarChartProps, 'slots'> {
  area?: boolean;               // fill under the line, color-mix to 18% transparent
}
export interface DoughnutChartProps {
  labels: string[];
  values: number[];
  slots?: CatSlot[];            // defaults to slots 1..N in order
  valueFormatter?: (value: number) => string;
}
```

Passing both `tone` and `slot`/`slots` is a development-time warning and `tone` wins — the collision rule made enforceable.

### Rendering

Values carried over from DAMA where they still apply: `ChartCard` extends the card surface, column flex, small gap, title as an uppercase muted label, canvas wrapper becomes an SVG wrapper at `position:relative; height:280px; width:100%`. Bars `rx: 4` anchored to the baseline (the dataviz skill's 4px rounded data-end spec, not DAMA's 6). Lines 2px. Points ≥ 8px. A 2px surface gap between adjacent bars.

### Interaction and accessibility

Scope the catalog does not contemplate, required by the dataviz skill and the real cost of choosing SVG:

- **Hover layer by default** — crosshair + tooltip on line, per-mark tooltip on bar and doughnut. Hit targets larger than the mark.
- **Legend whenever there are ≥ 2 series**; a single series takes no legend box (the title names it). Identity is never color-alone.
- **One axis. Never a dual-axis chart.**
- `role="img"` with an accessible name; the underlying numbers reachable as a table.

## Components

Each is a quartet: `X.jsx`, `X.d.ts` (with `@startingPoint`), `X.prompt.md` (usage, examples, Do/Don't per H10), and an entry in the group's `*.card.html`.

### `StatCard` (display)

```ts
export interface StatDelta {
  value: string;                       // preformatted, e.g. "+12%"
  direction: 'up' | 'down';            // draws the arrow
  tone?: 'positive' | 'negative' | 'neutral';  // decides the color; default 'neutral'
}
export interface StatCardProps {
  label: string;
  value: string;                       // preformatted
  delta?: StatDelta;
  sub?: string;
  icon?: React.ReactNode;
}
```

Root extends the card surface, `min-height: 120px`, column flex, tight gap. Header row: `label` (uppercase small label token) and `icon` pushed apart; icon ~14px, `opacity: .6`, muted. Value: display font, large, `font-variant-numeric: tabular-nums`, no margin. Sub: small muted.

Delta pill: `align-self: flex-start`, fully rounded, `padding: 2px 8px`, `font-size: 12px`, `font-weight: 600`, **transparent background with a 1px border**:

- `neutral` (default) → `--border-strong` border, `--text-muted` content
- `positive` → `--success` border and content
- `negative` → `--danger` border and content

### `PageHead` (navigation)

```ts
export interface PageHeadProps { title: string; subtitle?: string; actions?: React.ReactNode; }
```

Root `display:flex; align-items:flex-start; justify-content:space-between`, gap 16px, bottom margin 20px. Below `--bp-sm`: column, stretch, actions full width. Title: h1 token. Subtitle: small muted, ~2px top margin, no bottom margin. Actions: horizontal flex, 8px gap, wraps.

`Shell.jsx` is refactored to compose `PageHead` in its header instead of duplicating the markup.

### `Spinner` (feedback)

```ts
export interface SpinnerProps {
  size?: 'sm' | 'md' | 'lg';
  tone?: 'accent' | 'gold' | 'neutral' | 'on-accent';
  label?: string;                      // accessible name; defaults to "Loading"
}
```

Uses the `injected` style-tag pattern `ProgressBar` already establishes for its keyframes. Honors `prefers-reduced-motion` (slower, not stopped — as `Skeleton` does). `role="status"` with an accessible name.

**On tones.** The catalog says to "reuse `ProgressBar`'s tone vocabulary so the two feel like one family", but the two vocabularies are not the same: `ProgressBar` ships `accent | gold | success | danger | info`, while a spinner needs `neutral` and `on-accent` (for use inside a filled button) and has no use for `success`. `accent` and `gold` are shared and resolve to the same tokens; `neutral` and `on-accent` are spinner-specific. `ProgressBar`'s status tones are deliberately not carried over — an indeterminate wait has no state to report, and a spinner tinted `--danger` would read as a failure that has not happened. Document the divergence in `Spinner.prompt.md`.

### `ThemeToggle` (forms)

```ts
export interface ThemeToggleProps { label?: (isDark: boolean) => string; }
```

Built on `IconButton`. Sun icon when dark is active, moon when light. **Sets `aria-pressed` to the current dark state** — the part most often missed. Reads and writes through `theme.js`; owns no theme state.

## Extensions

### `Table` — responsive mode

```ts
export interface TableColumn<T = any> {
  // …existing…
  mobileLayout?: 'row' | 'block';    // 'row' = label/value pair (default); 'block' = full width, no label
}
export interface TableProps<T = any> {
  // …existing…
  responsive?: boolean;              // default true
}
```

Below `--bp-md`, measured on the **container** via `useContainerWidth`, render one card per row instead of a table. Card list: column flex, 16px gap. Card: card surface, hairline border, small padding, column flex, tight gap. `row` columns: `space-between`, header as an uppercase muted label on the left, value right-aligned and allowed to shrink (`min-width: 0`). `block` columns (typically actions): full width, right-aligned, top hairline divider, ~8px top padding.

### `Input` — `date` / `time` / `datetime-local`

DAMA uses native date inputs, not a picker. Arena does not need a `DatePicker`; it needs `Input` to not fall apart on these types.

- Verify label, focus ring, error and valid states for `date`, `time`, `datetime-local`.
- Style `::-webkit-calendar-picker-indicator` for the dark theme — near-invisible on `--color-base-100` by default. A pseudo-element cannot be reached from an inline style, so this uses the `injected` style-tag pattern `ProgressBar` already establishes. This is the one sanctioned exception to "components carry no CSS"; it targets a vendor pseudo-element, not a class of ours.
- `Input.prompt.md` documents native date/time as the sanctioned approach and that a custom `DatePicker` is deliberately not part of Arena.

Remove the loose hex values from prose where they imply the palette is normative. Document the ramp with its measured numbers.
- **`guidelines/components-danger.html`** — record decision 3 and its reasoning: destructive affordances are rare and must not compete visually, so filled red stays exclusive to `ConfirmDialog`'s final confirmation; delta pills are outline for both signs.

- **`CLAUDE.md`** — two statements go stale in this change and must move with it: "`styles.css` does nothing but `@import` the five files in `tokens/`" (it becomes six, and `palette.css` is the skin while `colors.css` is the structure), and the architecture note that components carry no CSS (the `injected` style-tag exception for vendor pseudo-elements is now real and load-bearing for `Input`).
- **`CHANGELOG.md`** — log the change.

## Verification

- `scripts/validate-palette.mjs` — the dataviz validator, versioned in the repo, plus a small runner asserting the shipped ramp clears every gate against both surfaces. Decision 4 makes this load-bearing: whoever swaps the skin re-runs it instead of trusting their eye.
- Serve the repo (`python3 -m http.server 8000`) and confirm every new `*.card.html` renders in both themes — the demos `fetch()` their JSX, so `file://` will not work.
- The repo has no test runner and no build; verification is the validator plus rendering the specimen and demo pages.

## Constraints that apply throughout

From `CLAUDE.md`: English only. No emoji. No gradients (sole exception `Skeleton`'s shimmer). No raw hex in components — define the token, alias, then consume. No CSS classes; inline `style` objects reading custom properties, hover/focus/active via local `useState`; `Button.jsx` is the reference shape. `README.md` is the normative spec and moves in the same change as any token or convention.

## Out of scope — domain compositions

Not Arena components, per the catalog's own reasoning: `camera-scanner`, `qr-card`, `course-color-chip`, `group-select`. The only system requirement hiding among them was the categorical palette, which this spec delivers. The **white media plate** for QR bitmaps (a surface that stays white regardless of theme) is a real token question deferred with `Calendar` to the next minor — nothing in this minor needs it.
