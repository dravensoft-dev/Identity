# Token debt and the gate's blind spots — Design

**Status:** approved in design
**Date:** 2026-07-19
**Execution order:** plan 4.5 of 6 — between the token/geometry boundary and framework-layer parity
**Depends on:** `2026-07-18-4-token-geometry-boundary-design.md` (plan 4) — executed
**Blocks:** `2026-07-18-5-framework-layer-parity-design.md` (plans 5a and 5b). Both consume what this adds, and 5a's arbitrary-value task moves here.

## Problem

`bun run check` is green: `check-dimension-literals` reports "no bare literals under
frameworks/", `check-arbitrary-values` reports none. Both are telling the truth about
the rule they implement. An audit of the tree on 2026-07-19 found that the rule has
four holes and that the repertoire is missing values the layers genuinely need.

Three separate findings, and they are not the same kind of thing:

**1. Tokens that exist and are not used.** `--focus-width` and `--focus-offset` have
**zero consumers in the whole repository** while four components write the focus ring
by hand. The gold focus ring is a named pattern in `README.md`; the only token that
describes it is dead.

| Site | Writes | Token that already satisfies it |
|---|---|---|
| `forms/Input.jsx:49` (×3), `forms/Textarea.jsx:11` (×2), `forms/Select.jsx:14`, `navigation/SegmentedControl.jsx:46` | `'0 0 0 2px var(--gold-soft)'` | `--focus-width` |
| `forms/Switch.jsx:17` | `translateX(18px)` | `calc(var(--sp-1) * 4.5)` |
| `feedback/Dialog.jsx:17` | `translateY(8px)` (keyframes) | `var(--sp-2)` |
| `navigation/Menu.jsx:17` | `translateY(-4px)` (keyframes) | `calc(var(--sp-1) * -1)` |
| `feedback/Tooltip.jsx:28` | `translateY(-8px)` | `calc(var(--sp-2) * -1)` |
| `feedback/Spinner.jsx:22` | `SIZES = { sm: 14, md: 20, lg: 32 }` | `--icon-sm` (14, exact), `--sp-5`, `--sp-8` |
| `display/Avatar.jsx:2` | `SIZES = { xs: 24, sm: 32, md: 40, lg: 56 }` | see the `avatar` family below |
| `display/Avatar.jsx:19` | `` `max(calc(var(--sp-1) * 2), ${d * 0.28}px)` `` | the `max()`'s first arm is a token; the second is not |

**2. Four blind spots in the gate**, each verified against the real `scanValue`, not
inferred:

- **`boxShadow` and `transform` are not governed properties.** `PROPS` in
  `check-dimension-literals.mjs` lists 30 properties and neither is among them, so
  `'0 0 0 2px var(--gold-soft)'` and `'translateX(18px)'` are never judged. This is the
  whole of finding 1's first three rows.
- **A template-literal interpolation breaks digit-unit adjacency.** `UNIT_LITERAL`
  matches a number immediately followed by a unit; in `` `…${d * 0.28}px` `` the `}`
  sits between them, so the value passes even though `width` **is** governed. Verified:
  `scanValue('width', '`max(calc(var(--sp-1) * 2), ${d * 0.28}px)`')` returns null.
- **CSS injected as a string is never read as CSS.** Every `@keyframes` in the tree
  lives in `s.textContent = '…'`, so `translateY(8px)` in `Dialog.jsx` and
  `translateY(-4px)` in `Menu.jsx` are invisible to a scanner that looks for JS
  `prop: value` sites.
- **SVG presentation attributes use `=`, not `:`.** `fontSize="10"` and
  `strokeWidth="2"` in the four charts are JSX attributes; the declaration scanner
  requires a colon. Confirmed asymmetry: `scanValue('fontSize', "'16'")` **does** flag
  ("a bare number, not a token") — the value is catchable, the position is not.

**3. Values with no token at all.** Loop durations are the clearest gap: `.7s`
(Spinner, Button), `1.15s` (ProgressBar), `1.4s` (Skeleton), `8s` (Rotor), and `2.4s`
repeated in **four** files as the reduced-motion step, plus Rotor's `24s`. `--dur-*`
models UI transitions (120/220/420ms) and nothing else. Also: the doughnut's centre
label at `16px`, which no type token matches, and `Shell.jsx:17`'s `232px` sidebar.

## Design

### 1. A `calc()` over tokens is a derivation, not a literal

`CLAUDE.md` already states the rule: *"A dimension in a framework layer is a token **or
a derivation of tokens**."* The inline-style gate implements that faithfully — it
strips every `var()` and judges the remainder, so `calc(var(--sp-1) * 4.5)` passes
because `4.5` is a multiplier, not a dimension.

`check-arbitrary-values.mjs` does not. Its `TOKEN_REF` accepts only a bare
`var(--token)`, optionally behind a type hint, so **no derivation can appear in a
Tailwind bracket at all**. The two gates disagree about what a derivation is.

This spec makes them agree: a bracket may hold a `calc()` (or `min()`/`max()`/`clamp()`)
whose every operand is a `var(--token)`, a unitless number, or a value in a unit the
token layer does not model. A bracket holding a bare dimension is still a violation.

```
legal:    text-[length:calc(var(--avatar-md)*0.4)]
          shadow-[inset_0_calc(var(--bw-strong)*-1)_0_var(--crimson)]
          w-[calc(var(--container-max)-var(--layout-sidebar))]
illegal:  text-[13px]   duration-[200ms]   z-[900]   bg-[#b52a20]
```

Two consequences worth stating, because both are the point rather than side effects:

- **Avatar's ratio moves into CSS.** React scales the initials at `diameter * 0.4` and
  the status dot at `diameter * 0.28`, which is why `Avatar.jsx:fontSize:d * 0.4` sits
  in `EXEMPT`. With the `avatar` family below and derivations legal, both become
  `calc(var(--avatar-md) * 0.4)` and `calc(var(--avatar-md) * 0.28)` — exact, in CSS,
  one source of truth, expressible in a manifest, and **the exemption is deleted rather
  than duplicated into the Angular layer**.
- **Tabs' underline follows the token.** Plan 5b currently carries three fallback
  spellings for `inset 0 -2px 0 var(--crimson)` because every one of them was illegal.
  One survives: `shadow-[inset_0_calc(var(--bw-strong)*-1)_0_var(--crimson)]`.

### 2. The four blind spots, closed

- **Govern `boxShadow` and `transform`.** Both already work under the existing
  `scanValue`: `translate(-50%,-100%)` passes on the free-unit list, `scale(0.98)`
  passes because a bare number must be the *entire* value to fail, and
  `rotate(120deg)` passes on `deg`. Only real dimensions start failing.
- **Judge each interpolation.** Before scanning, replace every `${…}` with a marker
  that keeps a following unit adjacent to a digit, so `${d * 0.28}px` reads as the
  dimension it renders. The interpolated expression itself is then scanned as its own
  value, which is what catches a ratio hiding in a template string.
- **Read injected CSS as CSS.** When a string literal assigned to `textContent` (or
  passed to `insertRule`) parses as CSS, run its declarations through the same
  `scanValue`, reusing `scripts/lib/css-decls.mjs` — the parser the drift gate and the
  Overview already share. This is the only new *machinery* in the plan; everything else
  is a list entry or a rule tweak.
- **Scan the attribute form.** Add the SVG presentation attributes the charts use —
  `fontSize`, `strokeWidth`, `strokeDasharray`, `r`, `cx`, `cy`, `x`, `y`, `width`,
  `height` — read in `prop="value"` position as well as `prop: value`. Attributes whose
  value is an expression binding are out of scope, exactly as they are in JS.

**Each fix is written test-first, and each one must first reproduce the miss it closes.**
A gate that has never been shown to fail is a gate nobody has tested.

### 3. Three additions to the repertoire, and one new step

**`loop` — cyclical motion**, a sibling of `dur` in `tokens/src/effects.json`, `$type:
duration`, values in `ms` to match `dur`'s existing shape. The values are exactly what
the components use today; nothing changes timing.

| Token | Value | Consumers today |
|---|---|---|
| `loop.spin` | 700ms | Spinner, Button |
| `loop.sweep` | 1150ms | ProgressBar (indeterminate) |
| `loop.shimmer` | 1400ms | Skeleton |
| `loop.brand` | 8000ms | Rotor |
| `loop.reduced` | 2400ms | Spinner, Button, ProgressBar — one value, copied in three files |
| `loop.brand-reduced` | 24000ms | Rotor |

`dur` stays what it is: the transition scale. A loop and a transition are different
roles, and merging them would repeat the `fs`/`dz` mistake the type scale already
retired.

**`avatar` — the four diameters**, `$type: dimension`: `xs` 24px, `sm` 32px, `md` 40px,
`lg` 56px. **This is the first family named after a component**, and the tension is
real enough to write down rather than gloss: `dz` and `icon` are named after roles, not
scales, so the layer already admits role-named families — but "avatar" names a
component. It earns its place because the ratio in §1 needs a token to derive from, and
because these four numbers are otherwise a JS map no gate can reach. If a second
component ever needs the same ladder, the family should be renamed for the role
(`figure`, say) rather than duplicated.

**`--layout-sidebar` — 232px**, beside `--container-max` and `--gutter` in
`tokens/src/spacing.json`. One consumer (`ui_kits/console/Shell.jsx:17`), and it is
example code rather than layer code, which is precisely why it is worth tokenizing:
the example is what a consumer copies.

**`dz.text-lg` — 16px.** The doughnut's centre percentage renders at 16px and no type
token matches: `--sp-4` and `--icon-md` are 16px but neither is a font size, and
`--fs-lg` is 17px. A data readout is chrome, not prose, so it belongs to `dz` — the
control scale that already runs 14/13/12/11/10. Adding a step above `dz.text` moves no
pixel; snapping to `--fs-lg` would have been a 1px design change, and this spec does
not make design changes.

### 4. The charts, half of them

Twelve sites in the four charts have an exact token and become
`style={{ … 'var(--token)' }}` instead of an attribute:

| Attribute | Sites | Token |
|---|---|---|
| `fontSize="10"` | BarChart, LineChart | `--dz-text-2xs` |
| `fontSize="11"` | BarChart, LineChart | `--fs-xs` |
| `fontSize="16"` | DoughnutChart | `--dz-text-lg` (new, §3) |
| `strokeWidth="1"` | BarChart ×2, LineChart ×2 | `--bw` |
| `strokeWidth="2"` | LineChart ×2, DoughnutChart | `--bw-strong` |

That also gives `--bw-strong` real consumers in the chart family, and it is a pure
conversion: every value stays what it is.

**The other half stays.** `CHART_HEIGHT` (280), `PAD` (8/8/28/44), the bar radius
`barPath(…, 4)`, the point radii `r={hover ? 5 : 4}`, the doughnut's `0.62`/`0.34` and
its `180`/`120` legend clamp, and `strokeDasharray="3 3"` are **arguments to JS
arithmetic**, not CSS. Tokenizing them would require a Style Dictionary target that
emits JS — the layer's rule is that nothing reads a custom property back into JS — and
that is a build-surface decision this spec declines. The charts keep the exception the
parity spec already grants them.

## Non-goals

- **A JS/TS token target.** Named above. It is the only way to tokenize chart geometry
  and it is out of scope; do not smuggle in a partial version.
- **Any change to a token value.** Every token this spec adds carries the value its
  consumers already render. `check-ramp.mjs`, `check-text-contrast.mjs` and the
  specimen pages should all be indifferent to this plan.
- **`ch`, `vw`, `vh`, `%` as tokens.** DTCG 2025.10 admits only `px` and `rem` in a
  `dimension`, so `42ch`, `92vw`, `12vh` and `62%` are not expressible. They stay
  literals under the shared unmodelled-unit list, which this plan moves out of plan 5a
  and lands here.
- **The exempt ratios.** `0.4`, `0.28`, `0.62`, `0.34` are multipliers. The rule
  governs dimensions, not the multiplier that derives one from another — and after §1
  the first two are no longer exemptions at all, because they are written in CSS.
- **Structural percentages.** `100%`, `50%`, `translate(-50%,-100%)`, Skeleton's
  `220% 100%` and `±140%`, ProgressBar's `40%` — composition geometry and animation
  shape, not design values.
- **Rotor's size.** The brand mark stays un-themeable and keeps its three `EXEMPT`
  entries. That reasoning is unchanged and is not reopened here.

## Verification

- `bun run check` passes — now including the widened gate.
- **Each blind spot has a test that fails before its fix.** Four reproductions:
  a governed `boxShadow`, an interpolated `${…}px`, a `@keyframes` string, and a
  `fontSize="10"` attribute.
- `grep -rn "0 0 0 2px" frameworks/` returns nothing; `var(--focus-width)` has four
  consumers where it had zero.
- `EXEMPT` loses `Avatar.jsx:fontSize:d * 0.4` and keeps the rest. A stale entry fails
  the gate, so this is self-enforcing.
- `bun scripts/check-dtcg.mjs`, `check-tokens-generated.mjs`, `check-ramp.mjs` after the
  token edits; `check-tailwind-coverage.mjs` must place every new token
  (`loop.*`, `avatar.*`, `layout-sidebar`, `dz.text-lg`) on a utility or in `EXCLUDED`
  with a reason, in the same task that adds it.
- **By eye, and this is the one that matters:** every animation runs at the same speed
  it does today (Spinner, Button, ProgressBar, Skeleton, Rotor, and each one's
  reduced-motion variant), the focus ring on Input/Textarea/Select/SegmentedControl is
  unchanged, Avatar's initials and status dot are unchanged at all four sizes, and the
  four charts are pixel-identical against `charts.card.html` as it stands on `main`.

## Affected files

**Token source:** `tokens/src/effects.json` (`loop`), `tokens/src/spacing.json`
(`avatar`, `layout-sidebar`, `dz.text-lg`) and their generated CSS; `tokens/src/TYPE-MAP.md`.

**Gates:** `scripts/check-dimension-literals.mjs` (`PROPS`, interpolation, injected CSS,
attribute form, `UNMODELLED_UNITS` export, one fewer `EXEMPT` entry),
`scripts/check-arbitrary-values.mjs` (derivations and unmodelled units),
`scripts/check-tailwind-coverage.mjs` (`EXCLUDED`), plus each one's test file.

**React** — this plan does touch it, unlike 5a and 5b: `forms/{Input,Textarea,Select,Switch}.jsx`,
`navigation/{SegmentedControl,Menu}.jsx`, `feedback/{Dialog,Tooltip,Spinner,ProgressBar}.jsx`,
`display/{Avatar,Skeleton}.jsx`, `brand/Rotor.jsx`, `charts/{Bar,Line,Doughnut}Chart.jsx`,
`ui_kits/console/Shell.jsx`.

**Docs:** `frameworks/tailwind/README.md` (what a bracket may hold), `CLAUDE.md` (the
dimension rule now names derivations in brackets too), `CHANGELOG.md` under `[Unreleased]`.

**Unchanged, explicitly:** every `tokens/palette*.json` value, `tokens/colors.css`,
`styles.css`, the plugin manifests, `support.js`, `theme.js`, `jsx-loader.js`.

Both parity plans are already written and need these edits once this lands:

- **5a Task 2 is deleted** — the unmodelled-unit fix moves here, with the other three
  blind spots and the derivation rule. 5a drops to 23 tasks and gains this as a
  prerequisite.
- **5a's Avatar task** loses its "two deliberate departures" note: the manifest reads
  `calc(var(--avatar-*) * 0.4)` and `* 0.28`, exact rather than snapped to the type
  scale, and adds no exemption.
- **5a's `animations.css`** reads `var(--loop-*)` instead of literal durations, which
  removes the one comment in either plan that admitted a literal.
- **5b Task 15 (Tabs)** collapses from three candidate spellings to one.

Two decisions taken alongside this spec, recorded here because they change the same two
plans: **`ChartCard` gains a manifest** (36 manifests, and the charts' exception narrows
to the three SVG charts), and **`Onboarding` implements `anchorRect`** in Angular with
`W`/`EDGE` as plain numbers and two `EXEMPT` entries mirroring React's — `Math.min`/`max`
need real numbers, and nothing in this layer reads a custom property back into JS.
