# The `@dsCard` viewport, and the gate that never measured it — Design

**Status:** DRAFT — not approved. Written 2026-07-20 from findings that surfaced during
plan 4.75's execution. The open questions at the end are real and must be answered before
a plan is written from this.
**Execution order:** plan 4.9 of 6 — after 4.75 (the four components), before 5a
(Angular primitive parity).
**Depends on:** `2026-07-19-4.75-applogo-sidenav-activityfeed-unauthcard-design.md`
(plan 4.75) — executed. Nothing else.
**Blocks:** `2026-07-18-5a-angular-primitive-parity.md` and
`2026-07-18-5b-tailwind-manifest-parity.md`, for the reason in *Sequencing* below —
both add specimen pages, and every page they add is born into whichever regime is in

## Problem

Every specimen and demo page in this repo opens with an HTML comment that declares the
size it should be rendered at:

```html
<!-- @dsCard group="Components" viewport="700x460" name="Navigation" subtitle="…" -->
```

That number drives external card rendering. **Nothing checks it against the page.** It is
a claim about a rendered result, written by hand, and no gate has ever compared the two —
so a page whose content outgrew its declared height goes on declaring the old one, and the
card that gets rendered from it silently loses whatever falls past the fold.

This is not hypothetical, and it is not rare. A measurement of **all 34 pages carrying a
`@dsCard` comment**, taken on 2026-07-20 at `0687d8c` with headless Chromium against the
page served over HTTP at its own declared width, found **nine that over-run what they
declare**:

| Page | Declares | Content measures | Over by |
|---|---|---|---|
| `frameworks/react/components/charts/charts.card.html` | `900x760` | `900x1345` | **+585h** |
| `frameworks/react/components/display/table-avatar.card.html` | `760x700` | `760x1151` | **+451h** |
| `frameworks/react/components/forms/forms.card.html` | `700x420` | `700x660` | **+240h** |
| `frameworks/react/components/display/display.card.html` | `700x460` | `700x533` | +73h |
| `frameworks/react/ui_kits/console/index.html` | `1280x820` | `1280x868` | +48h |
| `frameworks/react/components/feedback/alert.card.html` | `640x360` | `640x403` | +43h |
| `frameworks/react/components/feedback/feedback.card.html` | `700x480` | `700x516` | +36h |
| `guidelines/components-danger.html` | `700x350` | `700x383` | +33h |
| `guidelines/colors-categorical.html` | `700x200` | `700x218` | +18h |

The measurement was taken twice, with identical results both times. The remaining
twenty-five pages fit exactly — in every one of them, content height equals declared
height to the pixel, which is what a page that was measured when it was written looks
like.

`charts.card.html` is the sharpest case: it declares 760px and renders 1345px. **Nearly
half the chart specimens are below the fold of their own card** — and charts are the
components whose specimens most need to be seen whole, because a chart clipped at the
axis is not a smaller chart, it is a wrong one.

### Why this went unnoticed for so long

Because the failure is invisible from inside the repo. `bun run check` runs eight gates
and three test suites and reports the tree clean, correctly — none of them opens a page.
The pages themselves render fine when a human scrolls them in a browser; the loss happens
only in the external renderer, which crops to the declared box. So the one place the
defect is visible is the one place nobody looks during development.

### The evidence that this is a gate problem, not an authorship problem

Plan 4.75 shipped four components and touched four card pages. **Two of the four shipped
clipping, and both were caught only because someone happened to measure:**

- `navigation.card.html` gained a `SideNav` specimen. Its declared `700x460` was left
  alone; a task reviewer rendered the page and found the `BulkActionBar` specimen entirely
  below the fold. Raised to `700x640`.
- `brand.card.html` gained `AppLogo` at all four sizes. Its declared height was raised
  from `160` to `560` **by arithmetic** during authoring. The final whole-branch review
  measured it: `732x642` against a declared `780`-less `700x560` — clipping in *both*
  axes, with the `xl` lock-up (708px wide, the specimen the spec called the one that most
  needed seeing) running off the right edge. Raised to `780x660`.

Both slipped through per-task reviews that were otherwise clean and thorough. The second
one slipped through an authoring pass that had *already been told* the first one had
happened. That is the signature of a missing check rather than of carelessness: the
information needed to get it right is not available at the moment the number is written,
and no amount of care supplies it.

The four pages plan 4.75 touched all now measure exactly their declared size. The nine
above are what the same measurement finds everywhere else.

## Design

### 1. The gate measures; it does not guess

A new check — `bun run check:cards`, joining the eight in `check-all.mjs` — loads every
page carrying a `@dsCard` comment at its declared width, measures the rendered content,
and fails when the content exceeds the declared box in either axis.

It must **measure a real render**, not estimate. The pages transpile JSX in the browser
through `jsx-loader.js` and `fetch()` their own sources, so their height is not derivable
from the HTML: it depends on fonts, on the token layer, on React actually running. A gate
that parsed the markup would be a gate that reports clean while the tree is not, which is
the exact failure this spec exists to close.

That makes this **the first gate in the repo that needs a browser and a running server**,
and that is the whole cost of the proposal. Everything else here is small.

### 2. The tolerance is zero, and the pages are exact

The twenty-five clean pages measure their declared size *exactly* — not "within a few
pixels". So the gate has no reason to admit a tolerance, and admitting one would only
create a band in which a page may lie a little.

The gate should also flag a page that under-runs its declaration by a wide margin: a card
declaring 700x600 around 700x150 of content renders a mostly-empty box, which
misrepresents the specimen in the other direction. **That is a warning, not a failure** —
an over-tall card loses nothing, the reader sees the whole specimen plus empty space,
where a clipped one loses content silently. Only clipping fails the build.

### 3. The nine are fixed by measurement, in the same change

A gate added in one commit and its violations fixed in the next leaves the tree red in
between. The nine pages above get their declared viewports corrected to their measured
content, in the same change that adds the gate — the discipline plan 4.5 established for
every gate it widened.

**No page's content changes.** This spec moves declared numbers to match rendered reality;
it does not re-lay-out a single specimen. If a page turns out to need its content changed
rather than its declaration — `charts.card.html` at 1345px tall may genuinely want to be
two cards rather than one very long one — that is a design decision, and it is out of
scope here. Record it and leave the page taller.

### 4. `console/index.html` is probably not a card, and the gate must decide

The console is the one entry in the table that is not a specimen. It is a full example
application with its own scrolling content area, and a `+48h` over-run there may be
entirely legitimate — an app that scrolls is an app, not a clipped card.

Two ways out, and this needs deciding rather than defaulting: either the gate excludes
pages whose `group` is not `Components`, or `index.html` stops carrying a `@dsCard`
comment at all, since what it is for is being opened and clicked rather than rendered to
a card. The second is cleaner and the first is less disruptive.

## Non-goals

- **No content re-layout.** Declared numbers move to match what renders. Splitting
  `charts.card.html`, trimming `table-avatar.card.html`, or re-flowing `forms.card.html`
  are separate design decisions, each of which should have to argue for itself.
- **No new specimen pages.** 5a and 5b add those.
- **No visual-regression testing.** This gate answers one question — does the content fit
  the declared box — and deliberately not the much larger question of whether the content
  is *right*. A screenshot-diffing harness is a different proposal with a different cost.
- **No change to the `@dsCard` comment's format.** It gains no fields.
- **No gate for the pages that carry no `@dsCard` comment.** `Arena - Overview.html` and
  `Dravensoft Identity.dc.html` are not cards and are not in scope.

## Open questions

1. **Is a headless browser an acceptable dependency for `bun run check`?** It is present
   in the development environment used on 2026-07-20, and Playwright's Chromium is
   cached — but the repo's gates have been deliberately runtime-portable (they run under
   both Bun and node), and this one cannot be. It would be the first gate that simply does
   not run in some environments. What should it do there — skip loudly, or fail? **This is
   the question that decides whether the gate is worth building**, because a gate that
   silently skips is the same class of thing as the missing check it replaces.
2. **Is `console/index.html` a card?** See §4.

### Settled while this spec was being written

- **The cost question is closed.** An earlier draft asked whether ~2 minutes of rendering
  could survive a `bun run check` run before every commit. It cannot, and it does not have
  to: `CLAUDE.md` now states that the full sweep is expected **once, when a plan's
  implementation is finished**, with the individual gates staying available per commit.
  That is what makes a gate of this weight affordable at all.
- **Under-running is a warning, not an error.** A card declaring far more height than its
  content misrepresents the specimen, but it loses nothing — the reader sees the whole
  thing plus empty space. Report it; do not fail on it. Only clipping fails.
- **The gate owns its server.** Every other gate in `scripts/` is self-contained and this
  one should be too: it starts the static server on a free port, measures, and stops it,
  rather than requiring `bun run demos` to already be running. A gate with a prerequisite
  a human must remember is a gate that gets skipped.

## Sequencing

```
4.5  token debt + gate blind spots       <- executed
4.75 AppLogo/SideNav/ActivityFeed/UnauthCard  <- executed
     THIS SPEC (4.9)                     <- the viewport becomes machine-checked
5a   Angular primitive parity

5.5  chart geometry token target         <- draft, unapproved
6    four-package build + publish
```

It goes before 5a for the same reason 4.5 went before it. **5a adds a specimen page per
primitive — twenty-one of them — and 5b adds more.** Every page written before this gate
exists is a page whose declared height was chosen by eye and never verified, and the
measurement above says that is a coin flip: nine of the thirty-four pages written that way
are wrong, and two of the four written most recently, by people who had been warned, were
wrong too.

Landing the gate first means those twenty-one pages are born measured. Landing it after
means adding twenty-one more unverified declarations to a list this spec would then have
to be re-run to clean up.
