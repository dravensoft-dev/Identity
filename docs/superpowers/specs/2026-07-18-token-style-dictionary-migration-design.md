# Token migration to Style Dictionary (DTCG) ‚Äî design

**Date:** 2026-07-18
**Status:** Approved, ready for implementation
**Scope:** Sub-project 1 of 2. Sub-project 2 (the four npm packages: build + publish
pipeline) is a separate spec that depends on this one.

## Problem

Arena's design tokens are hand-authored CSS. `styles.css` `@import`s six files in
`tokens/`, and `tokens/palette.css` is declared, in prose across `CLAUDE.md` and
`README.md`, to be *the source of truth* ‚Äî the file a consumer swaps to re-skin
Arena.

The larger goal (sub-project 2) is to ship Arena as **four atomic npm packages**
‚Äî tokens, React, Angular, Tailwind ‚Äî one per integral Arena release, lockstep
versioned. For that to rest on *a complete, cross-platform standard* rather than
an Arena-specific CSS convention, the token layer must be authored in a
platform-neutral format that any present or future framework target can consume
without re-interpreting hand-written CSS.

This sub-project makes a **DTCG (Design Tokens Community Group, W3C) JSON the
single source of token values**, generated to CSS by **Style Dictionary v4** (the
DTCG reference implementation), while changing nothing about how any existing
consumer ‚Äî the demos, the plugin, the copy-in kit, the check scripts ‚Äî behaves.

The design was verified against the current Style Dictionary v4 / DTCG
documentation: every Arena token category maps to a first-class DTCG `$type`
(`color`, `dimension`, `fontFamily`, `fontWeight`, `number`, `duration`,
`cubicBezier`, `shadow`), references/aliasing are standardized (`{group.token}`
with `outputReferences`), and multi-theme output via per-theme builds is the
established Style Dictionary idiom (the DTCG "modes" spec is still draft, so
build-level theming is current best practice).

## Goals

1. **JSON is the single source of token values.** `tokens/src/**/*.json`
   (DTCG format) is the only file a contributor edits to change a token value.
2. **Style Dictionary generates four of the six token CSS files** ‚Äî
   `tokens/palette.css`, `tokens/typography.css`, `tokens/spacing.css`,
   `tokens/effects.css` ‚Äî which become **committed build output**, not
   hand-authored.
3. **Zero behavioral change.** The generated CSS declares the same custom
   properties, with the same values, under the same selectors, in the same
   cascade order. Demos, plugin, copy-in kit, and both check scripts keep working
   unchanged. A golden test proves declaration-set equality against today's files.
4. **The standard is made explicit and complete** via two reinforcements that
   ship as normative docs, so a future framework author has a doubt-free base:
   - **A documented DTCG type-map** (¬ßB): which `$type` every token group uses.
   - **A written layer contract** (¬ßC): what is standardized (the DTCG values)
     vs. what each platform re-implements (the runtime derivations).
5. **The boundary is honest.** `tokens/colors.css` (aliases + `color-mix`
   derivations + `var()` fallbacks) stays hand-authored ‚Äî DTCG has no
   color-composition type ‚Äî and `tokens/fonts.css` stays generated by
   `scripts/fetch-fonts.mjs` (DTCG has no `@font-face` type). Both are documented
   as the deliberate per-platform composition layer, not gaps.

## Non-goals

- **No value changes.** Not one token's computed value moves. This is a
  representation change (CSS ‚Üí JSON+generated-CSS), not a redesign.
- **No npm packaging.** The four packages, `tsup`/`ng-packagr`, publish, CI,
  versioning, and the `@dravensoft/*` scope are all sub-project 2.
- **No changes to `frameworks/*`**, the demos (`*.dc.html`, `*.card.html`,
  `guidelines/`), the plugin manifests, `support.js`, `theme.js`, `jsx-loader.js`,
  or `assets/`.
- **No change to `tokens/colors.css` semantics** ‚Äî it keeps its exact aliases and
  derivations; it merely keeps referencing the (now generated) `--color-*` vars.
- **No re-authoring of `tokens/fonts.css`** or its `fetch-fonts.mjs` generator.
- **No adoption of DTCG draft "modes"** ‚Äî theming stays at the build level.

## Design

### A. Source of truth ‚Üí `tokens/src/**/*.json` (DTCG)

New authored source, one concern per file, DTCG `$value`/`$type`:

```
tokens/src/
  palette.dark.json       ‚Üí emitted under  :root
  palette.light.json      ‚Üí emitted under  .arena-light
  typography.json         ‚Üí emitted under  :root
  spacing.json            ‚Üí emitted under  :root   (incl. density base dz-*)
  density.compact.json    ‚Üí emitted under  .arena-compact   (dz-* overrides)
  effects.json            ‚Üí emitted under  :root
  TYPE-MAP.md             ‚Üí the documented type-map (¬ßB), versioned with source
```

Group-level `$type` inheritance is used (Style Dictionary's `typeDtcgDelegate`
applies it), so a group declares `$type` once. Example (`typography.json`):

```json
{
  "fs": {
    "$type": "dimension",
    "display": { "$value": "64px" },
    "h1":      { "$value": "44px" },
    "md":      { "$value": "15px" }
  },
  "lh": {
    "$type": "number",
    "tight": { "$value": 0.98 },
    "body":  { "$value": 1.6 }
  }
}
```

The `--` prefix and the exact custom-property names (`--fs-display`, `--lh-body`,
`--color-cat-1`, ‚Ä¶) are produced by the CSS format's name transform (¬ßD), so the
generated names match today's byte-for-byte.

### B. Reinforcement 1 ‚Äî the documented DTCG type-map

`tokens/src/TYPE-MAP.md` is the normative table stating the `$type` of every
token group. It is the contract a new platform target reads first. Summary:

| Token group | Source file | DTCG `$type` | Notes |
|---|---|---|---|
| Base neutrals, brand, status, `error-fill`, `cat-1..8` | `palette.*` | `color` | per-theme (dark/light) |
| Font families (`font-display/body/mono`) | `typography` | `fontFamily` | comma stacks preserved |
| Font weights (`fw-*`) | `typography` | `fontWeight` | numeric 400‚Äì900 |
| Font sizes (`fs-*`) | `typography` | `dimension` | px |
| Line heights (`lh-*`) | `typography` | `number` | unitless |
| Letter spacing (`ls-*`) | `typography` | `dimension` | em |
| Spacing scale (`sp-0..24`) | `spacing` | `dimension` | px; `sp-0` = `0` |
| `container-max`, `gutter` | `spacing` | `dimension` | px |
| Breakpoints (`bp-sm/md/lg`) | `spacing` | `dimension` | px; read by JS via `getComputedStyle` |
| Density (`dz-*`) | `spacing` / `density.compact` | `dimension` | px; base + `.arena-compact` mode |
| Radius (`r-xs..pill`) | `effects` | `dimension` | px; `r-pill` = `999px` |
| Border widths (`bw`, `bw-strong`) | `effects` | `dimension` | px |
| Shadows (`shadow-1..3`) | `effects` | `shadow` | composite (incl. negative spread, rgba) |
| `glow-accent` | `effects` | `shadow` | references `--crimson-strong` (structural) ‚Äî see ¬ßC |
| `scrim` | `effects` | `color` | rgba |
| `scrim-blur`, `focus-width`, `focus-offset` | `effects` | `dimension` | px |
| Durations (`dur-fast/mid/slow`) | `effects` | `duration` | ms |
| Easings (`ease-*`) | `effects` | `cubicBezier` | `[x1,y1,x2,y2]` |

Tokens that are **not** in this map are, by definition, part of the structural
composition layer (¬ßC) ‚Äî they live in `colors.css` or `fonts.css`, never in
`tokens/src/`.

### C. Reinforcement 2 ‚Äî the written layer contract

Stated normatively in `README.md` and `CLAUDE.md`:

> **Standardized (the DTCG layer).** Every token *value* ‚Äî colors, dimensions,
> font attributes, durations, easings, shadows ‚Äî is authored once in
> `tokens/src/**/*.json` in DTCG format. This is the platform-neutral contract.
> A new framework target consumes this JSON (directly, or through a Style
> Dictionary platform that emits CSS / JS / iOS / Android / SCSS). Nothing here
> is Arena-specific.
>
> **Per-platform (the composition layer).** Two things DTCG deliberately does not
> model, and that therefore live in each platform's own idiom:
> 1. **Runtime color derivations** ‚Äî the muted-text levels and *-soft accents,
>    expressed in CSS as `color-mix(in oklab, var(--‚Ä¶) N%, transparent)` so they
>    re-derive when the skin swaps. In CSS these live in the hand-authored
>    `tokens/colors.css`. A new framework rebuilds this thin layer in its idiom
>    (e.g. Tailwind `color-mix` utilities, a JS token helper) **on top of the same
>    standard values** ‚Äî it never re-defines a value.
> 2. **`@font-face` bundling** ‚Äî generated by `scripts/fetch-fonts.mjs` into
>    `tokens/fonts.css`, pointing at the self-hosted `assets/fonts/` binaries.
>
> The dividing line: **DTCG owns values; the composition layer owns how values are
> combined at runtime.** `tokens/colors.css` therefore continues to hold *no skin
> value* ‚Äî only references (`var(--color-primary)`) and `color-mix` compositions.

`glow-accent` is the one `effects` token whose value embeds a reference into the
composition layer (`var(--crimson-strong)`). It is documented as such: stored in
`effects.json` with its color slot as a literal `var(--crimson-strong)` string,
rendered verbatim by the format. It is the single, named exception to "effects
values are self-contained."

### D. Build: `scripts/build-tokens.mjs` + custom CSS format

A programmatic Style Dictionary build (Node module, not just a `config.json`,
because it registers a format and loops themes):

- **One custom CSS format** wraps a file's tokens in the correct selector block
  and reproduces Arena's exact output ‚Äî property names (`--<group>-<name>`),
  value serialization (px kept as px, `cubicBezier` ‚Üí `cubic-bezier(...)`,
  `shadow` object ‚Üí `<x> <y> <blur> <spread> <color>`, `fontFamily` array ‚Üí
  comma stack), and `outputReferences` for any intra-JSON alias.
- **Theme/mode wiring** produces the multi-selector files by composing blocks:
  - `palette.css` = `palette.dark.json` under `:root` + `palette.light.json`
    under `.arena-light` (two blocks, one file ‚Äî matches today).
  - `spacing.css` = `spacing.json` under `:root` + `density.compact.json` under
    `.arena-compact` (matches today).
  - `typography.css`, `effects.css` = single `:root` block each.
- Each generated file opens with a header comment:
  `/* GENERATED by Style Dictionary ‚Äî edit tokens/src/, not this file. */`
  (mirrors the existing `tokens/fonts.css` generated-file convention).

Exact transform/format wiring (dimension serialization, DTCG `dimension`
object-vs-string handling, shadow rendering) is fixed in the plan and verified
against the Style Dictionary v4 docs; the golden test (¬ßH) is the arbiter.

### E. Generated CSS is committed

The four generated files are **committed to git**, not git-ignored. The plugin is
served *from the release tag* and the copy-in kit reads `tokens/*.css` directly;
both need the files present in the tree at the tag. This differs from
sub-project 2's package `dist/` (which will be git-ignored) ‚Äî token CSS is a
tracked, tag-frozen artifact. `node_modules/` is git-ignored (¬ßG).

### F. What stays hand-authored

- `tokens/colors.css` ‚Äî unchanged. Still `:root{--picker-invert:1}` /
  `.arena-light{--picker-invert:0}`, the alias block, and the `color-mix`
  derivations. It references the generated `--color-*` vars exactly as today.
- `tokens/fonts.css` ‚Äî unchanged; still generated by `scripts/fetch-fonts.mjs`.
- `styles.css` ‚Äî unchanged; the six `@import`s and their order are identical.

### G. Build tooling ‚Üí the repo's first `package.json`

A **dev-only, private** `package.json` at the repo root ‚Äî the change that retires
the "no package.json" identity:

```json
{
  "name": "arena",
  "private": true,
  "type": "module",
  "devDependencies": { "style-dictionary": "^4" },
  "scripts": {
    "build:tokens": "node scripts/build-tokens.mjs",
    "check:tokens": "node scripts/check-tokens-generated.mjs"
  }
}
```

- `private: true`, no `version`/`bin`/`exports` ‚Äî this package is never published
  (sub-project 2 introduces the four *published* package manifests, elsewhere).
- `node_modules/` is added to `.gitignore`.
- `style-dictionary` is the only dependency this sub-project adds.
- `style-dictionary-utils` (W3C token filters/formats) MAY be added during
  planning if it simplifies the shadow/`color-mix`-adjacent format work; noted,
  not committed to here.

### H. Verification scripts

- **`check-ramp.mjs` and `check-text-contrast.mjs` are unchanged.** They regex
  the `--color-*` / derivation declarations out of `tokens/palette.css` /
  `tokens/colors.css`; because the generated `palette.css` keeps identical
  declarations, both scripts run as-is ‚Äî and now double as a build-correctness
  gate on Style Dictionary's output.
- **New `scripts/check-tokens-generated.mjs`.** Runs the token build to a temp
  location and asserts the generated files' **declaration set** (the
  `--name: value;` pairs, per selector) equals the committed `tokens/*.css`.
  Exits 1 on any drift ‚Äî the guard against a committed file going stale relative
  to `tokens/src/` (the same "silent failure" concern that motivates
  `check-release.mjs`). Comments are not asserted (they migrate to `$description`
  best-effort); only declarations and their selectors are.

### I. Golden-file baseline

Before wiring Style Dictionary, snapshot the current four files' declaration sets.
After wiring, `check-tokens-generated.mjs` must report zero drift against that
baseline. This is the proof of Goal 3 (zero behavioral change): identical
`--prop: value;` under identical selectors, comments aside.

### J. Documentation to update (normative, same commit)

- **`CLAUDE.md`:**
  - "What this repo is" ‚Äî remove "no ‚Ä¶ package.json"; state a dev-only private
    `package.json` exists for the token build (still not a published npm package).
  - "Architecture ‚Üí Tokens are the only styling layer" ‚Äî rewrite: the source of
    token *values* is DTCG JSON in `tokens/src/`; `palette.css`, `typography.css`,
    `spacing.css`, `effects.css` are **generated** (edit the JSON + run
    `build:tokens`, never the CSS); `colors.css` remains the hand-authored
    structural/derivation layer holding no skin value; `styles.css` and
    `fonts.css` unchanged.
  - "When adding a color, define the daisyUI token in `palette.css` first‚Ä¶" ‚Üí
    define it in `tokens/src/palette.*.json` first, rebuild, then alias in
    `colors.css`. "After touching `palette.css` run `check-ramp`" ‚Üí after editing
    `tokens/src/` and rebuilding (`check-ramp` validates the generated
    `palette.css`); also run `check:tokens`.
  - Add the ¬ßC layer contract and a pointer to `tokens/src/TYPE-MAP.md`.
- **`README.md`:** wherever it names `palette.css` as the swap-to-reskin source of
  truth ‚Üí the DTCG JSON is the source; re-skinning edits `tokens/src/` and
  rebuilds. Add the layer contract (¬ßC) and type-map (¬ßB) as normative sections ‚Äî
  they are part of the design spec the README owns.
- **`CHANGELOG.md`:** one entry under `## [Unreleased]` (kept on top;
  `check-release.mjs` reads the first *versioned* entry).

### K. Release

Cutting a version is a **separate, user-triggered final step** (bump
`plugin.json` / `marketplace.json` / README header, rename the CHANGELOG heading,
pin `source.ref`, tag, `node scripts/check-release.mjs`) ‚Äî not part of this
change's implementation.

## Verification

- `npm run build:tokens` regenerates `tokens/{palette,typography,spacing,effects}.css`.
- `node scripts/check-tokens-generated.mjs` ‚Üí exit 0 (generated == committed,
  declaration-set equality; the golden baseline holds).
- `node scripts/check-ramp.mjs` ‚Üí exit 0 (both themes still clear every gate,
  reading the generated `palette.css`).
- `node scripts/check-text-contrast.mjs` ‚Üí exit 0 (unchanged).
- Serve over HTTP (`python3 -m http.server 8000`) and load a `guidelines/*.html`
  page, `Arena - Overview.dc.html`, and a component `*.card.html`: rendering is
  visually identical; DevTools shows the same computed custom-property values.
- `git diff --stat` on the four generated CSS files shows only header-comment and
  comment churn ‚Äî no `--prop: value;` pair changes.
- `grep`-level: no remaining prose in `CLAUDE.md`/`README.md` calls `palette.css`
  the hand-authored source of truth.

## Out of scope ‚Üí sub-project 2

The four atomic npm packages (`@dravensoft/arena-tokens` consuming this JSON +
generated CSS, `@dravensoft/arena-react` via `tsup`, `@dravensoft/arena-angular`
via `ng-packagr`, `@dravensoft/arena-tailwind` via `tsup`), lockstep versioning,
publish target, and CI. This spec deliberately delivers the standardized token
foundation those packages build on, and nothing more.
